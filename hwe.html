Portfolio Case Study

Hardware Events Data Pipeline & Analytics Dashboard

An end-to-end healthcare technology data pipeline for ingesting, normalizing, monitoring, and analyzing third-party hardware device events across remote care workflows.

Role Sole engineer and owner
Focus Backend, cloud infrastructure, ETL
Domain Healthcare hardware observability
Output Operational data lake and dashboard

Overview

The project created a reliable path from raw hardware events to analytics-ready data for operations and support teams.

I designed and implemented a hardware events pipeline that gives internal teams visibility into device battery health and cellular signal strength across third-party remote care hardware. The system receives vendor events, normalizes them into a shared model, stores operational records in DynamoDB, streams raw data into S3, transforms it through AWS Glue, and powers analytics dashboards through Athena and QuickSight.

Problem

Hardware health data existed, but it was not easy to monitor, compare, or operationalize.

Support and operations teams needed a dependable way to identify devices with low battery levels or poor cellular signal before those issues disrupted remote care workflows. The data arrived in vendor-specific formats, lacked a unified schema, and was not readily available for reporting, trend analysis, or proactive subscriber outreach.

What I Built

  • Built a TypeScript/NestJS AWS Lambda service to process batched SQS hardware events.
  • Integrated and normalized vendor-specific payloads for Anelto and Essence devices.
  • Designed a shared event schema for battery and cellular signal records.
  • Implemented DynamoDB persistence with composite keys and GSIs for organization, subscriber, device, and status access patterns.
  • Added local, Redis, API, and PostgreSQL lookup layers to resolve device relationships efficiently.
  • Built a Python Lambda triggered by DynamoDB Streams to flatten new records and write Parquet files to S3.
  • Created an AWS Glue ETL job to transform raw hardware events into curated fact and dimension tables.
  • Built Athena views joining event data with organization and subscriber context for dashboard reporting.
  • Configured QuickSight datasets and refresh patterns for operational analytics.
  • Managed infrastructure with Terraform and Serverless Framework, including API Gateway, SQS, DLQs, DynamoDB, IAM, Lambda, S3, and stream triggers.
  • Added observability through logs, retries, dead-letter queues, CloudWatch metrics, and Datadog instrumentation.
  • Wrote focused tests around mapping, caching, vendor strategies, DynamoDB behavior, and event processing.

Technical Decisions

The system was designed for scale, cost control, and long-term vendor extensibility.

  1. Decoupled ingestion from processing. API Gateway writes to SQS so event intake can absorb spikes independently from downstream Lambda processing.
  2. Processed events in batches. Lambda consumes SQS messages in batches to improve throughput and reduce database write pressure.
  3. Modeled DynamoDB around real access patterns. Composite keys and GSIs support organization, subscriber, device, and status-oriented queries.
  4. Separated operational storage from analytics. DynamoDB supports event processing while S3, Glue, Athena, and QuickSight power the analytics workflow.
  5. Used a strategy pattern for vendor events. Vendor-specific parsing is isolated so future hardware vendors can be added without rewriting the core pipeline.
